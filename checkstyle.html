Introduction

This project contains a customised Checkstyle configuration which attempts to walk the difficult line between "overly paranoid and won't even pass normal code" and "how the hell does this even work".

The configuration discussed below is open to debate and discussion. If there is anything you feel particularly strongly about, open a ticket.

Disabling checks

Checks can be ignored by using 'CHECKSTYLE.OFF: XXX' and 'CHECKSTYLE.ON: XXX' comments:

public boolean myMethod() {
    try {
        return doSomething();
    }
    // CHECKSTYLE.OFF: EmptyBlock
    catch (Exception e) {};
    // CHECKSTYLE.ON: EmptyBlock
    finally {
        return false;
    }
}

However, be warned that every time a check is ignored God kills two kittens.

Included modules

This configuration includes the following Checkstyle modules and configurations:

Annotations

The following sub-modules are used:

  • AnnotationUseStyle
  • MissingDeprectated
  • MissingOverride
  • PackageAnnotation
  • SuppressWArnings

Blocks

The following sub-modules are used:

  • EmptyBlock with configuration to allow commented empty blocks
  • LeftCurly with configuration to allow 200 character line lengths
  • NeedBraces
  • RightCurly with the configuration to force right curly braces to be on a new line:
    if (something) {
    // do stuff
} // <== note the position of this brace
    else {
// do other stuff
}
  • AvoidNestedBlocks

Design

The following sub-modules are used:

  • VisibilityModifier}
  • FinalClass}
  • InterfaceIsType}
  • HideUtilityClassConstructor}
  • DesignForExtension}
  • MutableException}
  • ThrowsCount} is configured to allow two checked exceptions to be declared
  • InnerTypeLast}

Coding

The following sub-modules are used:

  • CovariantEquals
  • EmptyStatement
  • EqualsAvoidNull
  • EqualsHashCode
  • FinalLocalVariable with configuration to explicitly check for final parameters
  • HiddenField with configuration to allow constructors and setters to hide the fields they set
  • IllegalInstantiation with configuration to exclude explicit creation of:
    • java.lang.Boolean
    • java.lang.String
  • InnerAssignment
  • MagicNumber with configuration to ignore values in annotations and hashcode calculations
  • MissingSwitchDefault
  • ModifiedControlVariable
  • RedundantThrows
  • SimplifyBooleanExpression
  • SimplifyBooleanReturn
  • StringLiteralEquality
  • NestedForDepth with configuration to allow two nested for loops
  • NestedIfDepth with configuration to allow two nested if statements
  • NestedTryDepth
  • NoFinalizer
  • SuperClone
  • IllegalCatch
  • IllegalThrows
  • PackageDeclaration
  • IllegalType
  • DeclarationOrder
  • ParameterAssignment
  • ExplicitInitialization
  • DefaultComesLast
  • FallThrough
  • MultipleStringLiterals with configuration to allow two uses of the same string literal
  • MultipleVariableDeclarations
  • OneStatementPerLine

Duplicates

The following sub-modules are used:
