Introduction

This project contains a customised PMD configuration which attempts to walk the difficult line between "overly paranoid and won't even pass normal code" and "how the hell does this even work".

The configuration discussed below is open to debate and discussion. If there is anything you feel particularly strongly about, open a ticket.

Disabling checks

Checks can be disabled by using PMD's standard suppression mechanism leveraging java's @Override annotation:

@SuppressWarnings("PMD.EmptyCatchBlock")
public boolean myMethod() {
    try {
        return doSomething();
    }
    catch (Exception e) {};
    finally {
        return false;
    }
}

However, be warned that every time a check is ignored God kills a kitten.

Included rulesets

This configuration includes the following PMD rulesets and configurations:

Basic

These rules are used as is, with one reconfiguration:

  • EmptyCatchBlock is reconfigured to allow documented catch blocks as some APIs we commonly use don't provide a meaningful check to execute first:
    try {
    return journalArticleLocalService.getArticleByUrlTitle(groupId, urlTitle);
}
catch (NoSuchArticleException e) {
    // Why is there no containsByUrlTitle() method?
}

Braces

These rules are used as is (and require the use of braces everywhere).

Clone

These rules are used as is.

Code Size

These rules are used as is.

Controversial

These rules are mostly used as is, with two exclusions:

  • OnlyOneReturn is excluded as it forces keeping lines of logic and variables alive longer for the life of the method - where that branch of code could exit cleanly straight away.
  • AvoidLiteralsInIfCondition is excluded as it disallows things like null checks such as: if (x == null) ... .

Coupling

These rules are used as is.

Design

These rules are used as is.

Finalizers

These rules are used as is.

Imports

These rules are mostly used as is, with one exclusion:

  • TooManyStaticImports is excluded as modern API design makes good and effective use of static methods and the readability that comes from staticly importing them. If you argue that it makes the source of those methods difficult to track, you should use an IDE with half-decent navigation!

J2EE

These rules are used as is.

JavaBeans

These rules are used as is.

Naming

These rules are used as is, with one reconfiguration:

  • LongVariable is reconfigured to allow variables with a length of up to 32 characters. We have wide monitors and auto-completion - why not use them?

Optimizations

These rules are used as is.

Strict Exceptions

These rules are used as is.

Strings

These rules are used as is, with one reconfiguration:

  • AvoidDuplicateLiterals is configured to allow duplicate values in annotations, to prevent warning on things like:
    @SuppressWarnings("PMD.something")
public void myMethod() {}

@SuppressWarnings("PMD.something")
public void myMethod() {}

Sun Security

These rules are used as is.

Type Resolution

These rules are used as is.

Unused Code

These rules are used as is.
